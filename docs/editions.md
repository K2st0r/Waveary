# Waveary Editions

[简体中文](./editions.zh-CN.md) · **English**

This page explains how the Waveary project is currently packaged and how future official offerings are expected to differ.

## Positioning

Waveary is an open source digital life companion framework.

The project should stay clear about one distinction:

- the open framework
- future official hosted or commercial offerings

That distinction helps users understand what is available today, what may arrive later, and what requires official cooperation.

## Editions Overview

| Edition | Status | Intended User | What It Means |
| --- | --- | --- | --- |
| `Waveary CE` | available now | developers, builders, early adopters | the open-source community edition |
| `Waveary Cloud` | future / planned | users who want an official managed service | a hosted official offering if launched later |
| `Waveary Enterprise` | future / planned | teams with stronger deployment, control, or compliance needs | a later enterprise-facing edition if it becomes necessary |

## Waveary CE

`Waveary CE` is the community edition.

Current direction:

- open-source codebase
- local or self-hosted usage
- bring-your-own model provider
- active framework and product-shell iteration

What CE is for:

- evaluating the framework
- building on top of the framework
- self-hosting
- experimenting with memory, relationship, timeline, voice, and companion-quality systems

What CE is not trying to be yet:

- a finished one-click consumer app
- an official managed cloud service
- a locked-down branded hosted subscription

## Waveary Cloud

`Waveary Cloud` is the name reserved for a possible future official hosted offering.

If launched later, it may include things like:

- official managed hosting
- easier onboarding
- fewer deployment steps
- cloud-managed continuity services
- more polished end-user operations

Today, this is a direction marker, not a public product commitment.

## Waveary Enterprise

`Waveary Enterprise` is the name reserved for a possible future enterprise-facing edition.

If it ever exists, it would likely focus on:

- private deployment needs
- admin and permission control
- compliance-sensitive environments
- deeper system integration

Today, this is also a direction marker, not an active commercial product.

## Open Source Code Versus Official Offerings

The current repository is the open project surface.

That means:

- the code license is open-source
- the official brand is reserved separately
- future official offerings, if any, are not defined by the code license alone

## Official Cooperation

If someone wants:

- official brand use
- official visual asset use
- co-branded distribution
- commercial partnership
- OEM-style packaging
- another custom arrangement

that should be handled as an official cooperation conversation, not assumed from the open-source code license.

## Current Practical Guidance

For most builders today, the clean path is:

1. use the open-source code under the repository license
2. self-host or build on top of it
3. rebrand if distributing your own product
4. contact the rights holder before using the official identity commercially
